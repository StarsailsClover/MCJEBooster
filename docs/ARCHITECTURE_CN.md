## 系统概述

MCJEBooster 是一个 JVM 级优化工具，将 Minecraft 的单线程 tick 处理转换为基于区域的多线程系统。它作为 Java Agent 运行，附加到正在运行的 Minecraft 进程并使用 ASM 进行字节码转换。

### 设计目标

1. **透明性**：无需修改 Minecraft 安装
2. **兼容性**：支持多个 Minecraft 版本和 Mod 加载器
3. **安全性**：失败时自动回滚
4. **性能**：在高负载场景下显著提升 TPS
5. **可维护性**：干净、文档完善、可测试的代码

## 核心组件

### 1. MCJEBoosterAgent

Java Agent 入口点，处理：
- Agent 初始化（premain/agentmain）
- 类转换器注册
- 版本检测
- 健康监控设置

### 2. InjectorMain

用于外部注入的独立应用程序：
- 使用 Java Attach API 进行进程发现
- 多进程场景的用户交互
- 向目标 JVM 加载 Agent

### 3. RegionScheduler

核心多线程引擎：
- ForkJoinPool 用于工作线程管理
- Z-order 空间分区
- 动态负载均衡
- TPS 计算和监控

### 4. MinecraftServerTransformer

基于 ASM 的字节码转换器：
- 方法签名检测
- 字节码模式匹配
- 多版本兼容性
- 具有回滚能力的安全转换

### 5. SyncPointManager

同步协调器：
- 用于 tick 同步的 CyclicBarrier
- 实体转移同步
- 方块更新排序
- 超时处理

### 6. VersionDetector

版本识别系统：
- 多方法检测（属性、类、字节码）
- 版本比较工具
- 映射感知检测

## 注入流程

```
1. 用户运行 InjectorMain
        │
        ▼
2. 扫描 Minecraft 进程
        │
        ▼
3. 选择目标进程
        │
        ▼
4. 通过 Attach API 附加到 JVM
        │
        ▼
5. 加载 MCJEBoosterAgent
        │
        ▼
6. Agent 初始化：
   a. 检测 MC 版本
   b. 注册转换器
   c. 转换已加载的类
   d. 初始化调度器
   e. 启动健康监控
        │
        ▼
7. 转换器修改：
   - MinecraftServer.tick()
   - ChunkProvider.tick()
   - Level.tickEntities()
        │
        ▼
8. 多线程 tick 开始
```

## 调度架构

### 区域分区

使用 Z-order（Morton）曲线将世界划分为区域以提高缓存效率：

```
不使用 Z-order：          使用 Z-order：
┌───┬───┬───┬───┐        ┌───┬───┬───┬───┐
│ 0 │ 1 │ 2 │ 3 │        │ 0 │ 1 │ 4 │ 5 │
├───┼───┼───┼───┤        ├───┼───┼───┼───┤
│ 4 │ 5 │ 6 │ 7 │        │ 2 │ 3 │ 6 │ 7 │
├───┼───┼───┼───┤        ├───┼───┼───┼───┤
│ 8 │ 9 │10 │11 │        │ 8 │ 9 │12 │13 │
├───┼───┼───┼───┤        ├───┼───┼───┼───┤
│12 │13 │14 │15 │        │10 │11 │14 │15 │
└───┴───┴───┴───┘        └───┴───┴───┴───┘
```

### 工作线程池

- 默认：CPU 数量 - 1 个工作者
- 使用 ForkJoinPool 进行工作窃取
- 每个工作者处理分配的区域
- 主线程通过屏障协调

### Tick 流程

```
主线程：
1. 获取读锁
2. 向工作者提交区域任务
3. 等待完成（带超时）
4. 释放读锁
5. 更新 TPS 计算

工作线程（每个区域）：
1. tick 区域内的实体
2. tick 区域内的方块
3. tick 区域内的方块实体
4. 记录执行时间
5. 信号完成
```

## 同步机制

### CyclicBarrier

- 在 tick 边界同步所有工作者
- 超时：45ms（为同步操作留出 5ms）
- 触发时自动重置
- 失败跟踪用于健康监控

### ReadWriteLock

- tick 期间的世界状态：读锁
- 防止并发修改
- 允许多个读者（工作线程）

### 实体转移同步

当实体在区域间移动时：
1. 源区域标记实体进行转移
2. SyncPointManager 协调交接
3. 目标区域接受实体
4. 原子更新以防止重复/丢失

## 性能考量

### 最佳场景

- 高实体数量（>1000）
- 大视距（>16）
- 红石装置
- 多玩家

### 次佳场景

- 空世界（收益最小）
- 低设置的单人游戏
- 受其他因素 CPU 限制

### 调优参数

| 参数 | 默认值 | 描述 |
|-----------|---------|-------------|
| workerCount | CPU-1 | 工作线程数量 |
| regionSize | 16 区块 | 每个区域的大小 |
| tickTimeout | 45ms | 最大 tick 持续时间 |
| rebalanceInterval | 100 ticks | 负载重平衡频率 |

## 安全与保障

### 注入安全

- 不对 Minecraft 进行永久修改
- Agent 可以分离
- 保留原始类用于回滚

### 健康监控

持续监控：
- TPS（必须保持在 5.0 以上）
- 线程存活
- 死锁检测
- Tick 超时

### 自动回滚触发器

| 条件 | 阈值 | 动作 |
|-----------|-----------|--------|
| 低 TPS | < 5.0 持续 5 ticks | 回滚 |
| 死锁 | 任何检测 | 回滚 |
| Tick 超时 | 连续 3 次 | 减少工作者 |
| 调度器死亡 | 立即 | 回滚 |

### 回滚过程

1. 停止 RegionScheduler
2. 恢复原始类定义
3. 禁用转换器
4. 记录回滚事件
5. 继续使用原版行为
